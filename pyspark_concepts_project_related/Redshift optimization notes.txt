1. DISTKEY — Data Distribution Across NodesWhat is it?
Controls which Redshift compute node stores which rows. When two tables have the same DISTKEY, matching rows land on the same node — no network shuffling during JOINs.Types:DISTKEY(column_name) — Key distribution, most common

Use when you frequently JOIN on that column
Rows with same column value always go to same node
DISTSTYLE ALL — copies entire table to every node

Use when table is small (less than few million rows)
Use when table is rarely updated
Use when table is frequently JOINed with large tables
Examples — date dimension, LOB lookup, currency codes
DISTSTYLE EVEN — round robin across nodes

Use when no clear join column exists
Use when table is rarely joined
DISTSTYLE AUTO — Redshift decides

Good default for new tables
Redshift analyzes query patterns and chooses best strategy
Real Scenario 1 — WITHOUT DISTKEY (slow):
loss_data:      Node1: policy 1,4,7    Node2: policy 2,5,8
premium_data:   Node1: policy 2,6,11   Node2: policy 1,7,13

JOIN on policy_id:
policy 1 is on Node1 in loss_data
policy 1 is on Node2 in premium_data
Data must travel across network for every single row
10 million policies = massive network shuffle
Query takes 45 minutes 😱Real Scenario 1 — WITH DISTKEY(policy_id) (fast):
loss_data:      Node1: policy 1,4,7    Node2: policy 2,5,8
premium_data:   Node1: policy 1,4,7    Node2: policy 2,5,8

JOIN on policy_id:
policy 1 is on Node1 in BOTH tables
JOIN happens locally on each node
Zero network traffic
Query takes 3 minutes 🚀Golden rule:
If two large tables are frequently JOINed — give them the same DISTKEY columnReal Scenario 2 — Small lookup table WITHOUT DISTSTYLE ALL:
line_of_business (50 rows):
Node1: GL, WC
Node2: Auto, ADP
Node3: DOT, Property

loss_data on Node1 has Auto policies
Must fetch Auto from Node2 every time
Constant network calls for a tiny 50-row table 😱Real Scenario 2 — WITH DISTSTYLE ALL:
line_of_business copied to every node:
Node1: ALL 50 rows
Node2: ALL 50 rows
Node3: ALL 50 rows

Every node has complete lookup table locally
Zero network calls
JOIN is instant 🚀2. SORTKEY — Sort Order Within Each NodeWhat is it?
Controls how data is physically sorted within each node. Makes range queries faster by enabling zone map pruning — Redshift reads min/max values per block and skips entire blocks that don't match your filter.Types:COMPOUND SORTKEY(col1, col2) — most common

First column is most important
Best when queries mostly filter on leading column
Less VACUUM maintenance needed
INTERLEAVED SORTKEY(col1, col2) — rare

All columns equally important
More VACUUM maintenance needed
Use only when queries filter equally on all sort columns
Real Scenario — WITHOUT SORTKEY (slow):
Node 1 blocks (random order):
Block 1: dates 2022-05, 2024-02, 2021-08, 2023-11
Block 2: dates 2023-03, 2022-09, 2024-01, 2021-12
Block 3: dates 2024-03, 2022-01, 2023-07, 2021-05
...1000 blocks total

Query: WHERE valuation_date BETWEEN 2024-01-01 AND 2024-03-31
Redshift has no idea which blocks contain 2024 dates
Must scan ALL 1000 blocks
Query takes 20 minutes 😱Real Scenario — WITH SORTKEY(valuation_date) (fast):
Node 1 blocks (sorted by date):
Block 1:  2021-01 to 2021-06  — skip, outside range
Block 2:  2021-07 to 2021-12  — skip
Block 3:  2022-01 to 2022-06  — skip
Block 4:  2022-07 to 2022-12  — skip
Block 5:  2023-01 to 2023-06  — skip
Block 6:  2023-07 to 2023-12  — skip
Block 7:  2024-01 to 2024-03  — READ THIS ✅
Block 8:  2024-04 onwards     — stop here

Only reads 1 block out of 1000
Query takes 45 seconds 🚀This is called Zone Map Pruning
Redshift stores min/max per block. With SORTKEY it knows exactly which blocks to skip — like a book index!3. Complete Table Design Examplesql-- Large fact table

DELETE/UPDATE marks rows deleted but doesn't free space
New INSERTs break the SORTKEY order
Query planner needs fresh statistics after bulk loads
sql-- VACUUM — reclaims space + re-sorts data
VACUUM my_table;

-- VACUUM FULL — full resort + reclaim space, most thorough
VACUUM FULL my_table;

-- VACUUM SORT ONLY — only resort, don't reclaim space
VACUUM SORT ONLY my_table;

-- VACUUM DELETE ONLY — only reclaim deleted space, don't resort
VACUUM DELETE ONLY my_table;

-- ANALYZE — updates statistics for query planner
-- Always run after bulk loads!
ANALYZE my_table;When to run what:

After bulk INSERT — run ANALYZE
After DELETE/UPDATE — run VACUUM DELETE ONLY
After many new INSERTs that broke sort order — run VACUUM SORT ONLY
Scheduled maintenance — run VACUUM FULL then ANALYZE
5. Push Down Predicates in GlueWhat is it?
Tells Glue to only read specific S3 partitions — skips irrelevant data before it even loads into memory. Only works on partition columns — columns used in partitionBy() when data was written.python# WITHOUT push down predicate — reads everything
dyf = glueContext.create_dynamic_frame.from_catalog(
    database = "my_db",
    table_name = "orders"
)
df = dyf.toDF().filter("year = '2024'")
# Reads 1TB, uses only 100GB 😱

# WITH push down predicate — reads only what's needed
dyf = glueContext.create_dynamic_frame.from_catalog(
    database = "my_db",
    table_name = "orders",
    push_down_predicate = "(year == '2024' and month == '01')"
)
# Reads only 100GB directly 🚀

# Your project example
push_down_predicate = f"(valuation_date == '{target_date}')"
dyf = glueContext.create_dynamic_frame.from_catalog(
    database = "my_db",
    table_name = "loss_data",
    push_down_predicate = push_down_predicate

Full load — old way:
Every run reads ALL data, transforms ALL, writes ALL
Example: 18 hours every single day
Problem: 90% of data unchanged — wasted effort!

Incremental load — new way:
Every run reads only NEW or CHANGED data
Example: 8-9 hours
Only processes what's actually new!python# Method 1 — Glue Job Bookmarks
job = Job(glueContext)
job.init(args['JOB_NAME'], args)

dyf = glueContext.create_dynamic_frame.from_catalog(
    database = "my_db",
    table_name = "orders",
    transformation_ctx = "datasource"
)
# Glue remembers last processed position automatically

job.commit()  # saves bookmark for next run

# Method 2 — Date filtering via job parameter
valuation_date = args['valuation_date']
dyf = glueContext.create_dynamic_frame.from_catalog(
    database = "my_db",
    table_name = "orders",
    push_down_predicate = f"(valuation_date == '{valuation_date}')"
)sql-- Method 3 — Redshift watermark

Default Spark creates one file per partition
200 partitions = 200 tiny 5MB files on S3
Too many files = slow reads later 😱python# Solution — coalesce before writing
df.coalesce(10).write.parquet("s3://bucket/output/")
# Result: 10 clean files of ~100MB each ✅

# With partitioning
df.coalesce(5)\
  .write\
  .partitionBy("year", "month")\
  .mode("overwrite")\
  .parquet("s3://bucket/output/")

# Increasing partitions — use repartition (full shuffle)
df.repartition(10)

# Decreasing partitions — use coalesce (no shuffle, faster)
df.coalesce(5)Ideal file sizes for S3:

Too small — less than 10MB — too many files, slow reads
Ideal — 128MB to 512MB — best balance
Too large — more than 1GB — hard to parallelize reads
8. Memory AnchorsSame DISTKEY = same node = no network travel
Two large tables JOINed slowly? Give both same DISTKEY column — matching rows land on same node, zero shuffle.Sorted = skip blocks = fast
Date range filter scanning everything? Add SORTKEY on date — Redshift reads zone maps and skips irrelevant blocks entirely.Small table = give everyone a copy
Small lookup table causing network calls? Use DISTSTYLE ALL — every node gets complete copy, JOIN is instant.Merge before writing = fewer files
Too many tiny S3 files slowing downstream reads? Coalesce before write — fewer larger files, faster reads.Only read what's new = faster runs
